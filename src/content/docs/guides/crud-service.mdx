`tools.dynamia.domain.services.CrudService` is the single abstraction for data access in
Dynamia Platform. Every CRUD view, module page, action, and application service goes through
it instead of hand-rolled repositories or a raw `EntityManager`. Inject it, use it, and let the
framework worry about the JPA plumbing.

<Aside type="caution">
Do not inject `EntityManager`/`Session` directly in application services. It bypasses the
listener chain (`CrudServiceListener`), auditing, and caching hooks that `CrudService`
provides — this is a documented anti-pattern in the framework.
</Aside>

## Default implementation

The out-of-the-box implementation is JPA-based (`JpaCrudService`), auto-configured by
`@EnableDynamiaTools`. You never instantiate it yourself — inject the `CrudService` interface:

```java
//demo/CustomerAppService.java
package demo;

import org.springframework.stereotype.Service;
import tools.dynamia.domain.services.CrudService;

import java.util.List;

@Service
public class CustomerAppService {

    private final CrudService crudService;

    public CustomerAppService(CrudService crudService) {
        this.crudService = crudService;
    }

    public void activate(Long customerId) {
        Customer customer = crudService.find(Customer.class, customerId);
        customer.setActive(true);
        crudService.save(customer);
    }

    public List<Customer> findActiveCustomers() {
        return crudService.find(Customer.class, "active", true);
    }
}
```

If a module needs an alternate `CrudService` (a different transaction manager, a remote/REST
backend, etc.), register it as a named bean and point a `CrudPage` at it with
`new CrudPage("id", "Name", Customer.class, "myCustomCrudService")`.

## Basic operations

| Method | Purpose |
|---|---|
| `save(T entity)` | Create or update — dispatches based on whether `id` is set. This is what you use 95% of the time. |
| `create(T entity)` | Forces an insert. |
| `update(T entity)` | Forces a merge/reattach. |
| `find(Class<T> type, Serializable id)` | Find by primary key. |
| `load(Class<T> type, Serializable id)` | Find by primary key **and** eagerly load its relationship graph. |
| `delete(T entity)` / `delete(Class type, Serializable id)` | Delete an instance or by id. |
| `deleteAll(Class type)` | Bulk-delete every row of a type — use with caution. |
| `count(Class type)` / `count(Class type, QueryParameters params)` | Row counts, optionally filtered. |
| `exists(Class type, Object id)` | Existence check without loading the entity. |
| `reload(T entity)` | Refresh an already-loaded instance from the database. |

```java
Book book = new Book("Effective Java", author);
book = crudService.create(book);            // forces INSERT

book.setPrice(new BigDecimal("45.00"));
crudService.update(book);                    // forces UPDATE

Book same = crudService.save(book);          // INSERT or UPDATE, based on id
```

## Filtering with `find` and `QueryParameters`

For anything beyond "give me everything", pair `find`/`findReadOnly`/`count` with
`QueryParameters` — a fluent, JPQL-agnostic filter/sort/paginate builder.

```java
List<Book> published = crudService.find(Book.class,
        QueryParameters.with("category", category)
                .add("active", true)
                .orderBy("title", true)
                .paginate(20));
```

- `QueryParameters.with(name, value)` — start a new instance with one equality filter.
- `.add(name, value)` — chain more equality filters.
- `.orderBy(column, asc)` / `.orderBy(column)` — sort (ascending by default).
- `.paginate(pageSize)` / `.paginate(new DataPaginator(totalSize, pageSize, page))` — page the result set.
- `.addGroup(otherParams, BooleanOp.OR)` — combine a nested group of conditions with `AND`/`OR`.

Plain values become `=` (or `LIKE` for searchable strings). For richer conditions, wrap the
value with a `QueryCondition` from `QueryConditions`:

```java
import static tools.dynamia.domain.query.QueryConditions.*;

List<Book> results = crudService.find(Book.class,
        QueryParameters.with("title", like("effective"))
                .add("price", between(20, 60))
                .add("publishedDate", geqt(LocalDate.of(2020, 1, 1)))
                .add("status", in(BookStatus.ACTIVE, BookStatus.FEATURED))
                .orderBy("publishedDate", false));
```

`QueryConditions` covers `eq`, `like`, `startsWith`, `endsWith`, `gt`/`geqt`/`lt`/`leqt`,
`between` (numbers, dates, `LocalDate`/`LocalDateTime` ranges), `in`/`notIn` (varargs, `List`,
or a subquery), and `isNull`/`isNotNull` — each with an overload accepting a `BooleanOp` to mix
`AND`/`OR` logic within the same `QueryParameters`.

Use `findReadOnly` instead of `find` for report/read-only screens — it skips dirty-checking
overhead for entities you know you won't mutate.

## `findByFields` and `findByExample`

Two shortcuts for search boxes and autocomplete widgets, without writing JPQL:

```java
// Search "smith" across name and email, keeping any pre-set filters
List<Contact> matches = crudService.findByFields(Contact.class, "smith",
        QueryParameters.with("active", true), "name", "email");

// Search-by-prototype: non-null fields on the example become filters
Contact example = new Contact();
example.setCity("Bogotá");
example.setActive(true);
List<Contact> sameCity = crudService.findByExample(example);

// ... with paging and sorting
List<Contact> page1 = crudService.findByExample(example,
        new DataPaginator(20), new BeanSorter<>("name", true));
```

## Custom and reusable queries

For anything `QueryParameters` alone can't express, build a `QueryBuilder` and reuse it across
call sites:

```java
public List<Book> findLowStock(int threshold) {
    QueryBuilder query = QueryBuilder.select()
            .from(Book.class, "b")
            .where("b.stock <= :threshold");

    return crudService.executeQuery(query,
            QueryParameters.with("threshold", threshold));
}
```

`executeQuery` also accepts a plain JPQL string directly (`executeQuery(String queryText,
QueryParameters parameters)`) when you don't need the builder's composability.

Aggregate values — `count`, `sum`, `max`, `avg` — go through `executeProjection`, which returns
a single typed result instead of a list:

```java
BigDecimal totalRevenue = crudService.executeProjection(BigDecimal.class,
        "select sum(o.total) from Order o where o.status = :status",
        QueryParameters.with("status", OrderStatus.PAID));

Long activeCount = crudService.executeProjection(Long.class, query); // QueryBuilder overload
```

## Bulk updates without loading entities

When you need to touch many rows or a single field, skip the load-mutate-save round trip:

```java
// Update one field on one entity, straight to the database
crudService.updateField(book, "stock", 42);

// Bulk update: set a field on every row matching the filter
int updated = crudService.batchUpdate(Book.class, "featured", true,
        QueryParameters.with("category", category));

// Bulk update: several fields at once
int rows = crudService.batchUpdate(Book.class,
        Map.of("active", false, "archivedAt", LocalDateTime.now()),
        QueryParameters.with("stock", 0));

// Atomic counters (a counter is just a numeric field on the entity)
crudService.increaseCounter(book, "viewCount");
crudService.deacreaseCounter(book, "stock");   // note: the API spells it "deacrease"
```

<Aside type="tip">
`updateField`, `batchUpdate`, `increaseCounter`, and `deacreaseCounter` issue direct UPDATE
statements — they bypass `CrudServiceListener` callbacks and JPA's first-level cache. Reach for
them when you need throughput on large sets; reach for `save`/`update` when your
[CRUD listeners](/guides/crud-listeners) or [validators](/guides/validators) must run.
</Aside>

## Running outside the current transaction

`executeWithinTransaction`/`saveWithinTransaction` open a **new** transaction, independent of
whatever is currently active. This is the escape hatch for background jobs, actions that must
persist a log entry even if the surrounding operation later rolls back, or long-running
[actions](/guides/actions) that shouldn't hold a single transaction open for their entire
duration:

```java
public void recordAudit(Book book, String message) {
    crudService.executeWithinTransaction(() -> {
        AuditLog log = new AuditLog(book.getId(), message);
        crudService.save(log);
    });
}

// Shortcut for the common "just save this in its own transaction" case
crudService.saveWithinTransaction(new AuditLog(book.getId(), "reindexed"));
```

## Best practices

- Call `CrudService` from application `@Service` classes, not directly from controllers or
  heavy view logic — keep persistence and presentation separate.
- Prefer `QueryParameters` + `QueryConditions` over string-concatenated JPQL; it's safer against
  injection and reads declaratively.
- Reach for `batchUpdate`/`updateField`/counters when you're updating many rows or a single
  field and don't need listener callbacks; reach for `save`/`update` when you do.
- Keep entities validated with Bean Validation and framework `Validator<T>` implementations —
  `CrudService` does not replace validation, it triggers it.
- For advanced multi-step operations, combine `CrudService` with `executeWithinTransaction` and
  [CRUD listeners](/guides/crud-listeners) rather than open-coding transaction boundaries.
